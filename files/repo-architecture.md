# Repository Architecture and Rendering Model

This document explains how this repository is structured and how markdown content is transformed into the published site.

## 1. High-level layout

- `docs/`: Source content used by Zensical.
- `scripts/`: Python generators and validators.
- `site/`: Rendered output (generated).
- `files/`: Maintainer documentation and templates.
- `zensical.toml`: Site configuration, theme/features, and navigation.

### Key source folders under `docs/`

- `docs/all-training/`: One markdown file per course/workshop.
- `docs/events/`: Event-oriented pages (overview, upcoming, past).
- `docs/explore/`: Discoverability pages:
  - `bundles/`
  - `learning-paths/`
  - `scientific-domains/`
  - `tags/`
- `docs/stylesheets/extra.css`: Shared custom styles.

## 2. What is authored vs generated

### Authored manually

- Course files in `docs/all-training/*.md`.
- Definition pages for learning paths and bundles.
- Site config in `zensical.toml`.
- CSS in `docs/stylesheets/extra.css`.

### Generated by scripts

- `docs/events/upcoming.md`
- `docs/events/past-events.md`
- `docs/events/index.md`
- `docs/all-training/index.md`
- `docs/explore/tags/index.md` and category pages
- `docs/explore/scientific-domains/index.md` and domain pages
- `docs/explore/learning-paths/index.md` and detail pages
- `docs/explore/bundles/index.md` and detail pages
- Generated sections inside course files (for example `## Tags`, bundle backlinks)

## 3. Rendering pipeline

The main entrypoint is:

```bash
.venv/bin/python scripts/generate_all.py
```

Execution order is defined in `scripts/generate_all.py`:

1. `add_course_dates.py`
2. `add_course_levels.py`
3. `validate_course_dates.py`
4. `generate_all_training_index.py`
5. `generate_upcoming.py`
6. `generate_past_events.py`
7. `generate_scientific_domains.py`
8. `validate_learning_paths.py`
9. `generate_learning_paths.py`
10. `generate_bundles.py`
11. `generate_tags.py`
12. `generate_events_index.py`
13. `generate_course_tags.py`
14. `zensical build`
15. `check_internal_links.py`

If any step fails, the pipeline exits non-zero.

## 4. Script responsibilities

### Data normalization and validation

- `add_course_dates.py`: Parses `**Dates**:` or `**Date**:` from course body and injects `start_date`/`end_date` into frontmatter.
- `add_course_levels.py`: Moves level tags (`beginner/intermediate/advanced`) into `level:` and removes them from `tags`.
- `validate_course_dates.py`: Validates status/format/date consistency and reports warnings/errors.
- `validate_learning_paths.py`: Ensures learning-path definitions reference existing course slugs.

### Page generators

- `generate_all_training_index.py`: Builds grouped all-training catalog page with status/level/format/dates.
- `generate_upcoming.py`: Builds upcoming events page from date/status rules.
- `generate_past_events.py`: Builds past events page from date/status rules.
- `generate_events_index.py`: Builds event overview page from course metadata.
- `generate_scientific_domains.py`: Builds scientific-domain overview and per-domain pages.
- `generate_learning_paths.py`: Builds learning path overview/detail pages.
- `generate_bundles.py`: Builds bundle overview/detail pages and bundle backlinks in course files.
- `generate_tags.py`: Builds tag overview and category pages using tag-category mapping.
- `generate_course_tags.py`: Injects rendered badge HTML into each course `## Tags` section.

### Shared helpers

- `course_utils.py`: Frontmatter parsing, iterating course files, loading normalized course records.
- `link_utils.py`: Canonical link construction between generated pages.
- `bundles_utils.py`, `learning_paths_utils.py`: Load and normalize definition files.

## 5. Page rendering model

1. Source markdown + frontmatter is read by script layer.
2. Scripts update generated markdown files/sections in `docs/`.
3. Zensical renders `docs/` into `site/` HTML.
4. Internal link checker validates rendered links in `site/`.

This means many pages are derived views over course metadata, not hand-maintained pages.

## 6. Navigation

Navigation is explicit in `zensical.toml` (`[project].nav`).

When adding a brand new section/page that should appear in menus, add it to nav explicitly.

## 7. Generated markers and idempotency

Some scripts replace sections bounded by generated comments or fixed headings. Keep these patterns intact so scripts remain idempotent.

Examples:

- `## Tags` section in course pages
- `<!-- GENERATED:bundle-links:start --> ... <!-- GENERATED:bundle-links:end -->`

Avoid manual edits inside generated blocks; edit source metadata instead.
